# 課題: Bike Sharing Demand Model Tuning Sandbox を作る

このドキュメントは、実データを使って **機械学習モデルの性能改善プロセスを学ぶための課題仕様** です。

元の `Collectible Prediction Sandbox` は、架空データで「データ作成 → 特徴量 → ラベル → 学習 → 評価 → 可視化」の流れを理解する課題でした。

ただし、短期間で機械学習の理解を深めるには、以下の理由から実データを使う課題に差し替える方が効果的です。

- すでに正解データがある
- 学習データとテストデータに分けて、モデル性能をすぐ確認できる
- ベースラインから改善までの試行錯誤を体験できる
- 評価指標が悪化した理由を、データやグラフから考えられる
- Streamlitで「どの工夫が効いたか」を見せやすい

そのため、この課題では UCI Machine Learning Repository の **Bike Sharing Dataset** を使います。

---

## 1. 判断: 課題は差し替えるべきか

結論:

```text
主課題は差し替える。
ただし、現行のコレクタブル課題は捨てず、Phase 0 の導入課題として残す。
```

理由:

| 観点 | 架空コレクタブル課題 | Bike Sharing 実データ課題 |
|---|---|---|
| データの理解 | 自分で作った世界なので理解しやすい | 現実のノイズや季節性を考える必要がある |
| 正解データ | 自分でラベルを作る必要がある | `cnt` が正解データとして用意されている |
| 性能評価 | サンプル世界のルールを拾えたかを見る | 未知データへの予測性能を評価できる |
| モデル改善 | 効いて当然の特徴量になりやすい | 特徴量、分割、モデルの差が出やすい |
| 初学者の学習効果 | MLの流れを理解するのに向く | MLの試行錯誤を理解するのに向く |
| Streamlit表現 | 商品ごとの可視化に向く | モデル比較、誤差分析、特徴量重要度に向く |

学習順としては、以下を推奨します。

```text
Phase 0: 架空コレクタブルデータでMLの全体像を理解する
Phase 1: Bike Sharing実データでベースラインモデルを作る
Phase 2: 特徴量とモデルを変え、性能改善の過程を記録する
Phase 3: Streamlitで改善プロセスを見える化する
```

```text
Bike Sharing Demand Model Tuning Sandbox
```

## 4. 学ぶべきこと

この課題で学ぶ中心は「モデルを一度作ること」ではなく、**性能を測りながら改善すること** です。

| 優先 | 学ぶこと | 理由 |
|---:|---|---|
| 1 | 回帰問題 | 数値を予測する基本問題だから |
| 2 | train/test分割 | 未知データに対する性能を見るため |
| 3 | 時系列分割 | 未来データを学習に混ぜない練習をするため |
| 4 | 評価指標 | MAE, RMSE, R2 の意味を理解するため |
| 5 | ベースライン | 改善したか判断する基準を作るため |
| 6 | 特徴量エンジニアリング | モデル性能を上げる主な方法だから |
| 7 | モデル比較 | モデルによって得意不得意があることを体験するため |
| 8 | 誤差分析 | スコアだけでなく、外した理由を考えるため |
| 9 | 可視化 | 数値だけでは見えない傾向を把握するため |
| 10 | Streamlit | 試行錯誤を人に説明できる形にするため |

## 5. 予測するもの

この課題では、`hour.csv` を使って以下を予測します。

```text
ある日時・天気・曜日・季節における、1時間あたりのレンタル自転車利用数 `cnt`
```

例:

```text
2012-07-01 18:00
夏、日曜日、晴れ、気温が高い、湿度が低い
=> 何台くらい借りられるか？
```

目的変数:

| カラム | 意味 |
|---|---|
| `cnt` | casual + registered の合計利用数 |

注意:

```text
`casual` と `registered` は `cnt` の内訳なので、通常の特徴量には入れない。
```

`casual` と `registered` を特徴量に入れると、答えの一部をモデルに渡してしまいます。
これはリークです。

## 7. 作る特徴量

最初は、すでにあるカラムだけで始めます。

### 7.1 Step 1: 基本特徴量

| 特徴量 | 内容 |
|---|---|
| `season` | 季節 |
| `yr` | 年 |
| `mnth` | 月 |
| `hr` | 時刻 |
| `holiday` | 祝日 |
| `weekday` | 曜日 |
| `workingday` | 平日 |
| `weathersit` | 天気 |
| `temp` | 気温 |
| `atemp` | 体感気温 |
| `hum` | 湿度 |
| `windspeed` | 風速 |

使わないカラム:

| カラム | 理由 |
|---|---|
| `instant` | 単なる行番号 |
| `dteday` | そのままではモデルに入れにくい。後で日付特徴量に変換する |
| `casual` | `cnt` の内訳なのでリーク |
| `registered` | `cnt` の内訳なのでリーク |
| `cnt` | 目的変数 |

### 7.2 Step 2: 日付・周期特徴量

| 特徴量 | 内容 |
|---|---|
| `is_weekend` | 土日か |
| `month_sin`, `month_cos` | 月の周期性 |
| `hour_sin`, `hour_cos` | 時刻の周期性 |
| `is_commute_hour` | 通勤時間帯か |
| `is_night` | 深夜か |

時刻は 0時と23時が近い関係にあります。
そのため、単なる数値より `sin` / `cos` に変換すると、周期性を表現しやすくなります。

### 7.3 Step 3: 過去情報を使う特徴量

時系列データなので、過去の利用数を特徴量にできます。

| 特徴量 | 内容 |
|---|---|
| `cnt_lag_1h` | 1時間前の利用数 |
| `cnt_lag_24h` | 24時間前の利用数 |
| `cnt_lag_168h` | 1週間前の同じ時間帯の利用数 |
| `cnt_roll_mean_24h` | 直近24時間の平均利用数 |
| `cnt_roll_mean_168h` | 直近1週間の平均利用数 |

注意:

```text
未来の `cnt` を特徴量に入れてはいけない。
rolling特徴量を作るときは、必ず shift してから rolling する。
```

悪い例:

```text
今の行の `cnt` を含めて rolling mean を作る
```

良い例:

```text
1行過去にずらしてから rolling mean を作る
```

| 段階 | モデル | 目的 |


この順番で確認します。

## 9. 分割方法

この課題では、主に時系列分割を使います。

### 9.1 悪い分割

```text
全期間をランダムに混ぜて train/test に分ける
```

ランダム分割は練習としては簡単ですが、未来の傾向が学習側に混ざりやすくなります。

### 9.2 推奨分割

```text
2011年のデータで学習
2012年のデータでテスト
```

または:

```text
前半80%で学習
後半20%でテスト
```

実運用では、過去データで学習して未来を予測します。
そのため、この課題でも未来をテストデータに残します。

## 10. 評価指標

回帰問題なので、分類の accuracy ではなく、以下を使います。

| 指標 | 意味 | 初学者向けの見方 |
|---|---|---|
| MAE | 誤差の絶対値の平均 | 平均して何台くらい外したか |
| RMSE | 大きな誤差を重く見る指標 | 大外しをどれくらい減らせたか |
| R2 | どれくらい説明できたか | 1に近いほど良い。0以下ならかなり弱い |

この課題では、まず MAE を中心に見ます。

理由:

```text
「平均して何台外したか」が直感的にわかるため
```

## 13. 実験の進め方

以下の順番で進めます。

### Step 1: データを読む

やること:

- `hour.csv` を pandas で読む
- `dteday` を日付型に変換する
- 行数、列数、期間を確認する
- 欠損がないことを確認する
- `cnt` の分布を見る

理解すること:

```text
まずデータの中身を知らないと、モデルの良し悪しは判断できない。
```

### Step 2: EDAをする

やること:

- 時刻別の平均利用数を見る
- 曜日別の平均利用数を見る
- 天気別の平均利用数を見る
- 気温と利用数の関係を見る

理解すること:

```text
特徴量は、データを見て仮説を立ててから作る。
```

### Step 3: ベースラインを作る

やること:

- 学習データの平均 `cnt` を常に予測する
- MAE, RMSE, R2 を出す

理解すること:

```text
ベースラインがないと、モデルが本当に良くなったか判断できない。
```

### Step 4: 線形回帰を作る

やること:

- 基本特徴量だけで Linear Regression を学習する
- ベースラインと比較する
- 係数を見る

理解すること:

```text
線形モデルは理解しやすいが、時間帯のような非線形パターンは苦手な場合がある。
```

### Step 5: 特徴量を追加する

やること:

- `is_weekend` を追加する
- `hour_sin`, `hour_cos` を追加する
- `month_sin`, `month_cos` を追加する
- ラグ特徴量を追加する

理解すること:

```text
モデルを変える前に、特徴量で改善できることが多い。
```

### Step 6: モデルを比較する

やること:

- Ridge を試す
- Random Forest を試す
- HistGradientBoostingRegressor を試す
- 同じ分割、同じ評価指標で比較する

理解すること:

```text
モデル比較は、条件をそろえないと意味が薄い。
```

### Step 7: 誤差分析をする

やること:

- MAEが大きい日を探す
- 時刻別に誤差を見る
- 天気別に誤差を見る
- 実測値と予測値を時系列で重ねる

理解すること:

```text
スコアが悪いときは、どこで外したかを見る。
```

### Step 8: Streamlitで見える化する

やること:

- 実験条件をUIで切り替える
- 評価指標を表示する
- グラフを表示する
- 実験履歴を表に残す

理解すること:

```text
モデル改善の過程を説明できる形にすると、理解が深まる。
```

## 15. 注意点

### 15.1 リークを避ける

入れてはいけない特徴量:

| カラム | 理由 |
|---|---|
| `casual` | `cnt` の一部 |
| `registered` | `cnt` の一部 |
| 未来の `cnt` | 予測時点では分からない |
| テスト期間全体から作った集計特徴量 | 未来情報が混ざる可能性がある |

### 15.2 ランダム分割だけで満足しない

ランダム分割でスコアが良くても、時系列分割で悪くなることがあります。

この差を見ることも大事な学習です。

### 15.3 スコアだけを追わない

スコア改善は大事ですが、初学者の目的は以下です。

```text
なぜ良くなったか
なぜ悪くなったか
どの特徴量が効いたか
どの条件で外したか
```

これを説明できるようにします。
