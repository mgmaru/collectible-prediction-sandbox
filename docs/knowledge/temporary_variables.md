# 一時変数を置くかどうかの判断基準

作成日時: 2026-07-04 JST

## このドキュメントの目的

実装中に迷いやすい、次の疑問を整理する。

```text
処理結果を一時変数に置くべきか、
それともそのまま return や関数の引数に書くべきか
```

結論としては、**名前を付けることで意味が増えるなら、一時変数に置く**。

逆に、名前を付けても意味が増えない場合は、直接書いてよい。

## 基本方針

迷ったら、学習中は一時変数に置いてよい。

理由は、処理の途中に名前が付くことで、データの流れを追いやすくなるため。

今回のアプリでは、次のような流れがある。

```text
CSV全体
↓
選択商品の日次データ
↓
選択商品のイベントデータ
↓
グラフ
↓
画面表示
```

この流れに名前を付けると、コードを読むときに理解しやすくなる。

例:

```python
selected_item_data
selected_item_events_data
fig
event_date
event_type
```

## 一時変数を置くメリット

| メリット | 内容 |
|---|---|
| 意味が明確になる | `selected_item_data` のように、何のデータか名前で分かる |
| デバッグしやすい | 途中の値を `print` や `st.dataframe` で確認できる |
| 処理の流れが追いやすい | 上から順番に読める |
| 重複を減らせる | 同じ関数呼び出しを何度も書かなくてよい |
| 型チェックしやすい | 必要なら変数に型を付けられる |

## 一時変数を置くデメリット

| デメリット | 内容 |
|---|---|
| 変数名を考える必要がある | 悪い名前だと、逆に読みにくくなる |
| 変数が増えすぎる | どれが重要な値か分かりにくくなる |
| 使い回しすぎると危険 | 同じ変数の中身が途中で変わると追いにくい |
| 不要な一時変数はノイズになる | 名前を付けても意味が増えない変数は読み手の負担になる |

## 判断基準

### 一時変数に置いた方がよい場合

| 条件 | 理由 |
|---|---|
| 同じ結果を複数回使う | 重複を避けられる |
| 処理の意味に名前を付けたい | 読みやすくなる |
| デバッグで中身を確認したい | 途中状態を確認しやすい |
| 関数呼び出しが長い | ネストが深くならない |
| 後続処理の材料になる | データの流れが追いやすい |

### 直接書いてよい場合

| 条件 | 理由 |
|---|---|
| 1回しか使わない | 変数名を増やす必要がない |
| 処理が短くて明確 | 直接読んでも負担が少ない |
| 名前を付けても意味が増えない | ただの言い換えになる |

## 良い一時変数の例

```python
selected_item_data = extract_selected_item_data(
    selected_item,
    daily_snapshots_data,
)

selected_item_events_data = extract_events_for_selected_item(
    selected_item_data,
    events_data,
)
```

この例では、一時変数を置くことで次の意味が明確になる。

| 変数名 | 意味 |
|---|---|
| `selected_item_data` | 選択された商品の日次スナップショット |
| `selected_item_events_data` | 選択された商品のイベント一覧 |

この2つは、表の表示やグラフ描画など、後続処理で使われる。
そのため、一時変数に置くのが自然。

## 直接returnしてよい例

```python
def extract_selected_item_data(
    selected_item: str,
    items_data: pd.DataFrame,
) -> pd.DataFrame:
    return items_data[items_data["item_name"] == selected_item]
```

この関数では、処理が短く、戻り値の意味も関数名から分かる。

```text
extract_selected_item_data
-> 選択商品のデータを抽出する
```

そのため、関数内で必ずしも一時変数を置く必要はない。

ただし、次のように書いても悪くはない。

```python
def extract_selected_item_data(
    selected_item: str,
    items_data: pd.DataFrame,
) -> pd.DataFrame:
    selected_item_data = items_data[items_data["item_name"] == selected_item]
    return selected_item_data
```

学習中であれば、この書き方の方が処理の意味を確認しやすい。

## あまり良くない一時変数の例

```python
data = selected_item_data
result = fig
value = "date"
```

これらは、名前を付けても意味が増えていない。

| 変数名 | 問題 |
|---|---|
| `data` | 何のデータか分からない |
| `result` | 何の結果か分からない |
| `value` | 値の意味が分からない |

一時変数を置くなら、読み手が理解しやすい名前にする。

## 今回のアプリで一時変数に置くとよいもの

| 変数 | 一時変数に置く理由 |
|---|---|
| `daily_snapshots_data` | CSV全体の日次データとして何度も使う |
| `events_data` | CSV全体のイベントデータとして何度も使う |
| `selected_item_data` | 表示とグラフで使う中心データ |
| `selected_item_events_data` | イベント表とグラフ縦線で使う |
| `fig` | 作成後に軸設定やイベント線追加を行う |
| `event_date` | 縦線の位置として使う |
| `event_type` | 縦線のラベルとして使う |

特に `fig` は、一時変数に置くべき例。

理由は、次のような流れになるため。

```text
グラフを作る
↓
X軸の表示を調整する
↓
イベント日の縦線を追加する
↓
Streamlitで表示する
```

このように、作ったあとに操作を続ける値は、一時変数に置くと読みやすい。

## 判断ルール

迷ったときは、次の順番で考える。

```text
1. その値は後で複数回使うか
2. その値に意味のある名前を付けられるか
3. 途中で中身を確認したいか
4. 直接書くと長くなりすぎないか
```

1つでも当てはまるなら、一時変数に置いてよい。

最終的な判断基準:

```text
名前を付けることで意味が増えるなら、一時変数に置く
名前を付けても意味が増えないなら、直接書く
```

## 今回の結論

今回のような学習用アプリでは、一時変数を置く方針でよい。

理由:

- データの流れが見えやすい
- デバッグしやすい
- 後から見返したときに理解しやすい
- Streamlitやpandasの処理は途中状態を確認する場面が多い

ただし、変数名は重要。

悪い名前を付けるくらいなら、直接書く方が読みやすい場合もある。

一時変数は「行数を増やすため」ではなく、**処理の意味をコードに残すため**に使う。
